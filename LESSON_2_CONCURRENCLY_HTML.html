How we created threads

class MyThread extends Thread {
    @Override
    public void run() {
        System.out.println("work in " + getName());
    }
}

Thread t = new MyThread();
t.start();
Runnable job = () -> {
    System.out.println("work in "
        + Thread.currentThread().getName());
};

Thread t = new Thread(job, "worker-1");
t.start();
Reminder: in modern Java, passing a Runnable is usually preferred because it separates the task from the thread object.
🛡️

How we handled shared state

  • Protect methods used by multiple threads with synchronized.
  • Use thread-safe or concurrent collections when many threads touch the same structure.
  • Identify the critical section: the smallest block that must not interleave incorrectly.
public synchronized void increment() {
    count++;
}
Key transition: synchronized works, but it is a blunt tool. If you mark an instance method as synchronized, the lock is the whole object instance. That may be more locking than you really need.

Why java.util.concurrent exists

The package gives us more expressive tools so we can match the synchronization technique to the actual problem.

The java.util.concurrent.atomic package

Atomics are designed for shared values that need safe, lock-free updates. They are especially useful for counters, flags, sequence numbers, and simple state transitions.

⚛️

Main atomic classes

Class Typical use
AtomicInteger Counters, IDs, inventory counts
AtomicLong Larger counters and timestamps
AtomicBoolean On/off flags, cancellation markers
AtomicReference<T> Safely swap object references
Think of atomics as specialized tools for a small piece of shared state, not as a replacement for all locking.
🔁

Common operations

AtomicInteger count = new AtomicInteger(0);

count.incrementAndGet();
count.decrementAndGet();
count.addAndGet(5);
int value = count.get();
count.set(42);

boolean changed = count.compareAndSet(42, 43);
Big idea: compareAndSet lets you update the value only if it is still what you expected. That is the foundation of many lock-free algorithms.

Before: synchronized counter

class VisitorCounter {
    private int count = 0;

    public synchronized int next() {
        count++;
        return count;
    }
}

After: atomic counter

class VisitorCounter {
    private final AtomicInteger count =
        new AtomicInteger(0);

    public int next() {
        return count.incrementAndGet();
    }
}
✅

Great use cases

  • Simple counters
  • Sequence generators
  • One shared boolean flag
  • Replace one reference safely
⚠️

Not enough when

  • You must update multiple fields together
  • You need a bigger critical section
  • You must protect collection invariants
  • The operation is more than one shared value
🧠

Thinking shortcut

If the shared state is “just one value,” try atomic first. If the rule spans many values or a whole object state, think locks or concurrent collections.

The java.util.concurrent.locks package

Explicit locks give us control that synchronized does not directly expose, such as timed lock attempts, manual unlock points, and multiple condition queues.

🔒

Basic ReentrantLock pattern

private final Lock lock = new ReentrantLock();
private int count = 0;

public void increment() {
    lock.lock();
    try {
        count++;
    } finally {
        lock.unlock();
    }
}
Rule: always unlock in a finally block. Forgetting to unlock is a classic bug.
⚖️

synchronized vs Lock

Feature synchronized ReentrantLock
Syntax simplicity Very simple More verbose
Automatic unlock Yes No, must call unlock()
Try without waiting forever No direct equivalent tryLock()
Interruptible waiting Limited lockInterruptibly()
Multiple conditions One monitor wait set Condition objects

Useful lock features

tryLock() Ask for the lock and move on if it is not available right now.
tryLock(timeout, unit) Wait for a bounded amount of time instead of blocking forever.
lockInterruptibly() Useful when cancellation matters and a blocked thread should be able to give up.
Condition Explicit condition variables for workflows like “not empty” and “not full.”

When should you choose a lock?

  • When synchronized is too rigid.
  • When you need more control over when to wait and when to back off.
  • When you want clearer coordination around several conditions.
  • When using advanced producer-consumer patterns later.
For many everyday cases, synchronized is still perfectly valid. The point is not that locks are always better. The point is that they solve different problems.

ReentrantReadWriteLock: separate locks for readers and writers

A plain ReentrantLock lets only one thread in at a time — even if every thread just wants to read. ReentrantReadWriteLock fixes this by maintaining two coordinated locks: many readers may hold the read lock simultaneously, but a writer gets exclusive access.

📖

The core idea

Situation Allowed?
Multiple threads reading at the same time ✅ Yes — readers do not block each other
One thread writing while others read ❌ No — writer waits for all readers to finish
Two threads writing at the same time ❌ No — writers are mutually exclusive
Best when reads are frequent and writes are rare — a read-heavy cache, a configuration store, or a leaderboard that is read thousands of times per second but updated occasionally.
🔑

Basic pattern

private final ReadWriteLock rwLock =
    new ReentrantReadWriteLock();

private final Lock readLock  = rwLock.readLock();
private final Lock writeLock = rwLock.writeLock();

private Map<String, String> cache = new HashMap<>();

public String get(String key) {
    readLock.lock();
    try {
        return cache.get(key);   // many threads at once
    } finally {
        readLock.unlock();
    }
}

public void put(String key, String value) {
    writeLock.lock();
    try {
        cache.put(key, value);   // exclusive access
    } finally {
        writeLock.unlock();
    }
}

ReentrantLock vs ReentrantReadWriteLock

Feature ReentrantLock ReentrantReadWriteLock
Concurrent reads No — one thread at a time Yes — unlimited simultaneous readers
Exclusive writes Yes Yes
Complexity Simple — one lock object Two lock objects from one factory
Best when Reads and writes are equally common Reads vastly outnumber writes

Watch-outs

Writer starvation If readers never stop arriving, a waiting writer may never get the lock. Some implementations support a fair mode (new ReentrantReadWriteLock(true)) to prevent this.
No lock upgrading A thread holding the read lock cannot upgrade to the write lock without releasing first. Attempting this causes a deadlock.
Always unlock in finally The same rule as ReentrantLock — both the read lock and the write lock must be released in a finally block.
If you are not sure whether reads dominate, stick with a plain ReentrantLock. The extra complexity of a read-write lock only pays off when the read-to-write ratio is high.

Synchronizers: coordinating groups of threads

Sometimes the problem is not “protect one variable.” Sometimes the real problem is “how do several threads wait for each other or share limited access?”

🚦

The four synchronizers in this lesson

  • CountDownLatch — wait until a count reaches zero.
  • Semaphore — allow only a limited number of threads through.
  • CyclicBarrier — make a fixed group wait until everyone arrives.
  • Phaser — like a more flexible, multi-phase barrier.
🧭

Mental model

Tool Picture in your head
CountDownLatch A gate that opens once after N signals
Semaphore A room with only a few seats
CyclicBarrier A team meeting point at the end of each round
Phaser A reusable stage manager for several rounds

CountDownLatch

CountDownLatch latch = new CountDownLatch(3);

// worker threads call:
latch.countDown();

// coordinator thread calls:
latch.await();
  • Use it when one thread must wait for several other tasks to finish.
  • Typical examples: wait for startup, wait for three downloads, wait for test fixtures.
  • One-shot: once the count hits zero, it does not reset.

Semaphore

Semaphore slots = new Semaphore(2);

slots.acquire();
try {
    useSharedPrinter();
} finally {
    slots.release();
}
  • Use it when access should be limited, not exclusive.
  • Typical examples: 3 DB connections, 5 API tokens, 2 printers.
  • It models permits rather than ownership of one lock.

CyclicBarrier

CyclicBarrier barrier = new CyclicBarrier(3);

// each worker does part 1
barrier.await();
// now all three begin part 2
  • Use it when a fixed team repeatedly meets at the same checkpoint.
  • Common in simulations, matrix work, and round-based processing.
  • Reusable after the barrier is tripped.

Phaser

Phaser phaser = new Phaser(3);

phaser.arriveAndAwaitAdvance();
// phase 0 finished, move to phase 1
phaser.arriveAndAwaitAdvance();
// phase 1 finished, move to phase 2
  • Best when work happens in several stages.
  • More flexible than CyclicBarrier.
  • Can register or deregister parties as the workflow changes.
Class Best question it answers Resettable?
CountDownLatch “Has the required number of tasks finished yet?” No
Semaphore “How many threads may access this resource at once?” Permits are reused
CyclicBarrier “Has the whole team reached the checkpoint?” Yes
Phaser “What phase are we in, and has everyone completed it?” Yes, multi-phase

Thread pools and executors

Creating raw threads for every task is expensive, noisy, and hard to manage. A thread pool lets a fixed set of worker threads reuse the cost of thread creation and process many tasks.

❌

Raw-thread style

for (Task task : tasks) {
    new Thread(() -> process(task)).start();
}
  • Creates many thread objects
  • Hard to limit concurrency
  • Hard to shut down cleanly
  • Harder to collect results
✅

Executor style

ExecutorService pool =
    Executors.newFixedThreadPool(4);

for (Task task : tasks) {
    pool.submit(() -> process(task));
}

pool.shutdown();
  • Reuses worker threads
  • Controls concurrency level
  • Cleaner shutdown model
  • Works with both Runnable and Callable

Common executor ideas

Concept Meaning
Executor Very simple interface: just execute a task
ExecutorService Adds lifecycle management, submission, shutdown, futures
Fixed thread pool A stable number of workers
Task queue Submitted work waits until a worker is free

Why thread pools are better

  • You can bound the number of active threads.
  • You avoid creating a brand new thread for every tiny unit of work.
  • You centralize task execution.
  • You can reason about system load more easily.
  • You prepare students for server-side programming, where executors are everywhere.
A useful classroom line: threads are workers, executors are managers, and tasks are the jobs handed out to the workers.

Callable and Future

Runnable is great when the task just performs work. Callable is for tasks that return a value and may throw checked exceptions.

📦

Runnable vs Callable

Interface Method Returns value? Checked exceptions?
Runnable void run() No No
Callable<T> T call() Yes Yes
⏳

What is a Future?

A Future<T> is a handle to a result that may not be ready yet.

  • get() waits for the result
  • isDone() checks whether the task finished
  • cancel(...) requests cancellation
A future turns “start this in another thread” into “start this work and give me the answer later.”

Runnable submission

ExecutorService pool =
    Executors.newFixedThreadPool(2);

pool.submit(() -> {
    System.out.println("saving log entry");
});

Callable + Future

ExecutorService pool =
    Executors.newFixedThreadPool(2);

Future<Integer> future = pool.submit(() -> {
    return 21 + 21;
});

Integer answer = future.get();
Important Note: future.get() blocks. If you call it immediately after every submission, you may accidentally turn concurrent code back into mostly sequential code.

Let's talk

Why not just use synchronized for everything?
Because it does not communicate intent very well for all problems. A semaphore says “limited permits.” A latch says “wait for N completions.” A future says “result later.” Those are easier to reason about than one giant lock.
Do atomics replace locks?
No. Atomics are ideal for small independent shared values. If several fields must change together, or a whole data-structure invariant must be protected, you usually still need locking or a concurrent data structure.
Why are executors such a big deal?
Because modern applications usually think in terms of tasks, not hand-managed thread objects. Executors let us submit work, bound resources, and gather results more cleanly.

In-class exercises

  1. AtomicTaskCounter — Refactor a synchronized counter to use AtomicInteger (increment, decrement, reset) with no synchronized keyword.
  2. WorkerStartupLatch — Use a CountDownLatch to make a coordinator thread block until a fixed number of worker threads have finished startup.
  3. PrinterRoom — Use a Semaphore to model a shared printer room where at most N print jobs may run concurrently.
  4. BatchTaskExecutor — Submit a list of tasks to a fixed ExecutorService thread pool and verify that at most POOL_SIZE distinct worker threads are ever created.
  5. ParallelSumCalculator — Split a list of integers across N Callable<Long> workers, collect all Future handles first, then retrieve results — keeping work truly concurrent.
  6. IndependentCounters — Replace a coarse per-instance synchronized method with two separate ReentrantLock fields so that reads and writes do not block each other.
